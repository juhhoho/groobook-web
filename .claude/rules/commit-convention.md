## Commit Convention Rules
당신은 프로젝트의 히스토리를 정갈하게 유지하고, 변경 사항을 논리적 단위로 관리하는 전문 개발자 역할을 수행합니다. 모든 커밋과 푸시는 아래 규칙을 엄격히 준수하십시오.

1. 커밋 메시지 구조
모든 커밋 메시지는 다음 형식을 따릅니다.

Plaintext
[type]: [subject]

[description]
2. 타입(Type) 정의
변경 사항의 성격에 따라 가장 적합한 타입을 선택하십시오.

feat: 새로운 기능 추가

fix: 버그 수정

design: UI 스타일, 레이아웃 변경 (CSS, Tailwind 등)

refactor: 코드 로직 개선 (결과물은 동일하나 내부 구조 개선)

style: 코드 포맷팅, 세미콜론 누락 수정 (로직에 영향 없음)

docs: README, 주석 등 문서 수정

chore: 빌드 설정, 패키지 관리, 단순 잡무

test: 테스트 코드 추가 및 수정

3. 원자적 커밋 (Atomic Commits) 지침
논리적 분리: 사용자가 여러 작업을 한 번에 요청하더라도, 성격이 다른 작업(예: 로직 수정과 디자인 수정)은 반드시 별도의 커밋으로 분리하여 실행합니다.

커밋 시점: 하나의 '의도' 또는 '작업 단위'가 완료될 때마다 즉시 로컬 커밋을 수행합니다.

언어: 제목과 본문은 모두 한국어로 작성합니다.

본문: - 불렛 포인트를 사용하여 변경된 핵심 내용을 상세히 기술합니다.

4. 로컬 커밋 및 푸시 워크플로우
당신은 로컬에 쌓인 '푸시되지 않은 커밋'의 상태를 관리해야 합니다.

상태 추적: 현재 로컬 브랜치에 origin으로 푸시되지 않은 커밋이 몇 개인지 항상 인지합니다. (git cherry -v 등을 활용)

푸시 제안: 사용자의 요청(Task)이 일단락되었거나, 미푸시 커밋이 3개 이상 쌓인 경우 사용자에게 다음 사항을 확인합니다.

현재 로컬에 쌓인 커밋 목록 요약

"현재 로컬에 [N]개의 커밋이 있습니다. 이 작업들을 하나로 합쳐서(Squash) 푸시할까요, 아니면 개별 커밋 그대로 푸시할까요?"

Squash 실행: 사용자가 '합쳐서 푸시'를 선택할 경우, 해당 작업들을 대표하는 하나의 깔끔한 커밋으로 rebase 또는 squash 하여 원격에 반영합니다.

5. 예시 (Example)
사용자 요청: "로그인 API 연결해주고, 로그인 버튼 색깔 파란색으로 바꿔줘."

당신의 수행:

feat: 로그인 API 연동 및 예외 처리 커밋 생성

design: 로그인 버튼 테마 색상 변경 (Blue) 커밋 생성

질문: "로그인 로직과 디자인 수정을 마쳤습니다. 현재 2개의 커밋이 로컬에 있는데, 하나로 합쳐서 푸시할까요?"
